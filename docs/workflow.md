# Workflow

The workflow is intentionally staged:

1. Read local instructions.
2. Classify the request mode.
3. Clarify domain terms when needed.
4. Decide whether OpenSpec is required.
5. Create and validate OpenSpec artifacts when required.
6. Stop for approval before implementation.
7. Use Superpowers for approved implementation planning and execution.
8. Preflight Review the current Plan/Brief revision before execution or dispatch.
9. Apply Step Evidence Gate to complete business slices, not TDD micro-steps.
10. Run verification and Review; actionable findings return to fix, re-verify, and Review.
11. Persist fresh final verification evidence, then run final diff/scope Review.
12. Reconcile OpenSpec tasks, archive when appropriate, and validate after archive.
13. Report conclusion, evidence, residual risk, and next steps.

Use the lightweight path when the risk is low. Use the strict path when contracts or public behavior can change.
