# FableCodex and Caveman Review

## Conclusion

Do not build a three-layer parallel execution architecture with OpenSpec, FableCodex, and pi-caveman.

Recommended positioning:

```text
Main flow: openspec-superpower-change
Optional reference: FableCodex
Output style: pi-caveman / caveman-style
Knowledge sink: Obsidian or another long-term note system
```

## FableCodex

FableCodex concepts such as inspect-first, goal ledger, evidence checkpoint, findings gate, and verify-before-completion overlap with existing Step Evidence Gate, review-only mode, OpenSpec tasks, Superpowers plan, and Superpowers verification.

Use it only as:

- external checklist;
- review inspiration;
- evidence-gate comparison;
- personal research note;
- lightweight workflow for projects that do not use `openspec-superpower-change`.

Do not use it as a second execution layer for the same change.

## Caveman-style output

Caveman-style compression is an output style, not a governance mechanism.

Can compress:

- progress updates;
- final chat summaries;
- simple bugfix summaries;
- commit message drafts;
- daily Q&A.

Must not over-compress:

- OpenSpec proposal;
- design docs;
- spec deltas;
- Superpowers plans;
- Step Evidence Gate;
- verification records;
- risk and rollback notes;
- database, security, production, or customer-facing documents.
